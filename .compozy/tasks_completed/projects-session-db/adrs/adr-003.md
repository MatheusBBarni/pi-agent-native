# ADR-003: Use System SQLite With Local Store Wrapper

## Status

Accepted

## Date

2026-05-08

## Context

The PRD requires local project/session continuity, stale project visibility, stale project removal, and minimal session metadata. The current implementation persists all state through a static JSON `SessionStore`. The package currently has no SQLite dependency and only depends on MarkdownUI.

Technical clarification selected Apple system SQLite through `SQLite3`/`libsqlite3` with a small local wrapper. This keeps dependencies low while satisfying the SQLite persistence decision from ADR-001.

## Decision

Implement project/session persistence with the macOS system SQLite library through a small local Swift wrapper owned by `SessionStore`. The wrapper will expose the existing high-level `load` and `save` shape first, then add narrowly scoped operations needed by V1 such as stale project removal.

The SQLite store will use a local database file in the Pi Agent Native application support directory. It will store projects, sessions, selected project identity, and selected session identity. It will not introduce a broad repository framework or generalized persistence platform.

## Alternatives Considered

### Alternative 1: Add GRDB.swift

- **Description**: Use GRDB for typed records, migrations, and query helpers.
- **Pros**: Mature Swift SQLite ergonomics, safer query construction, built-in migration patterns.
- **Cons**: Adds a package dependency for a narrow persistence slice.
- **Why rejected**: The selected approach favors minimal dependencies and a small local wrapper.

### Alternative 2: Keep TechSpec Library-Agnostic

- **Description**: Specify SQLite behavior but leave the access layer choice to implementation.
- **Pros**: Maximizes implementation flexibility.
- **Cons**: Leaves a major technical decision unresolved for task generation.
- **Why rejected**: The TechSpec should produce implementable tasks with clear dependencies.

### Alternative 3: Keep JSON Persistence

- **Description**: Continue using `sessions.json` and avoid SQLite for V1.
- **Pros**: Smallest immediate code change.
- **Cons**: Conflicts with ADR-001 and does not provide the intended durable project/session identity model.
- **Why rejected**: SQLite was already accepted as the persistence substrate.

## Consequences

### Positive

- Avoids adding third-party dependencies.
- Keeps the implementation close to the current `SessionStore` boundary.
- Provides enough control over schema, file location, and write behavior for the V1 scope.

### Negative

- Requires careful manual SQLite statement handling.
- Tests must cover wrapper error handling and schema setup because there is no higher-level library.
- Future schema evolution will require explicit migration code if the persistence model grows.

### Risks

- Manual SQL code could become verbose or error-prone. Mitigation: keep the wrapper small, use parameterized statements, and cover the store with temporary-database unit tests.
- The wrapper could expand into a persistence platform. Mitigation: keep V1 limited to project/session continuity operations.

## Implementation Notes

- Keep the database path accessible through settings diagnostics, replacing or superseding the current JSON path display.
- Do not use this ADR to prescribe UI layout or business scope.
- The follow-on tasks should decide whether to import `SQLite3` directly or add the required system library linker setting in SwiftPM.
