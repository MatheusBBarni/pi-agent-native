# ADR-004: Keep Continuity State Fresh and Computed

## Status

Accepted

## Date

2026-05-08

## Context

The PRD says V1 starts fresh because the product has not launched and there are no real users with `sessions.json` data to migrate. It also requires stale projects to remain visible, stale project removal to affect only local app records, and sessions to show whether they can be resumed.

Technical clarification selected computed project availability, Pi RPC session-id-based resumability, and removal through `WorkspaceStore`/`AppModel` rather than direct persistence mutation.

## Decision

Do not migrate legacy `sessions.json` data. The new project/session store starts empty for first launch under this feature.

Represent project availability as a computed view of the stored project path rather than as a durable project status field. The app keeps unavailable projects in the normal project list and computes whether the path is currently available when rendering or restoring.

Represent session resumability for V1 as the presence of a Pi RPC session id. The app may still fail gracefully at switch time if Pi cannot restore that session.

Implement stale project removal as an `AppModel`/`WorkspaceStore` operation that removes the local project record and associated local sessions, then persists the resulting state. It must never mutate the filesystem project directory.

## Alternatives Considered

### Alternative 1: Migrate Existing JSON

- **Description**: Import `sessions.json` into SQLite on first launch.
- **Pros**: Preserves legacy local state.
- **Cons**: Adds migration work for a product that has not launched and has no real user data.
- **Why rejected**: The user clarified that migration is unnecessary.

### Alternative 2: Persist Project Availability Status

- **Description**: Store `available` or `stale` directly on the project record.
- **Pros**: Simple UI reads.
- **Cons**: Availability can change outside the app and stored status can go stale.
- **Why rejected**: Computed availability better reflects filesystem reality.

### Alternative 3: Remove Stale Projects in Persistence Directly

- **Description**: Delete stale project records in the store and reload UI state afterward.
- **Pros**: Keeps removal close to storage.
- **Cons**: Bypasses current AppModel/store ownership and makes UI state refresh more fragile.
- **Why rejected**: Current architecture centralizes user actions through AppModel and in-memory stores.

## Consequences

### Positive

- Avoids unnecessary migration complexity.
- Keeps stale project behavior accurate when files move or reappear.
- Keeps user actions inside existing app-state ownership boundaries.
- Makes stale project removal testable without filesystem mutation.

### Negative

- Existing local developer JSON state will not carry forward.
- Computed availability requires filesystem checks in view-model or store logic.
- Pi-session-id resumability may show a session as resumable before a missing session file is discovered by Pi.

### Risks

- Computed availability could become expensive if done repeatedly for large project lists. Mitigation: keep checks lightweight and bounded to visible project records.
- Resumability may be optimistic. Mitigation: show restore failures clearly and refresh session status after failed switch attempts.

## Implementation Notes

- Update or replace the current `sanitizePersistedProjects` behavior because it silently drops missing directories.
- Add stale project removal behavior through AppModel and WorkspaceStore.
- Keep archive and delete workflows out of V1.
