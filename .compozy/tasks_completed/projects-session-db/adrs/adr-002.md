# ADR-002: Scope PRD to Continuity MVP

## Status

Accepted

## Date

2026-05-08

## Context

The PRD discovery and clarification process identified a broader possible project/session lifecycle feature: restart continuity, stale project visibility, archive actions, and destructive delete actions inside configuration. The user selected the Continuity MVP approach for the PRD, which prioritizes reliable project/session restoration after restart over richer lifecycle management.

The codebase already has project selection, per-project session listing, last-session restore, and JSON-backed state persistence. The market research shows that recent projects and session restore are baseline expectations in IDEs and agent tools. The first PRD should therefore focus on making continuity dependable before expanding into full project/session management.

## Decision

Scope the PRD to the Continuity MVP:

- Restore known projects and sessions after app restart.
- Show unavailable/stale projects instead of silently hiding them.
- Provide a remove action for stale projects.
- Show minimal session metadata: title, status, last updated, and resumability.
- Start fresh under the new persistence model rather than requiring migration of old `sessions.json` data.

Archive project, archive session, delete project, and delete session are out of scope for this PRD.

## Alternatives Considered

### Alternative 1: Continuity + Archive

- **Description**: Add archive project/session actions on top of continuity.
- **Pros**: Gives users a non-destructive cleanup mechanism.
- **Cons**: Expands lifecycle semantics before restart reliability is validated.
- **Why rejected**: The selected PRD direction prioritizes restart continuity over management depth.

### Alternative 2: Managed Continuity

- **Description**: Add restart continuity, archive actions in the main experience, and permanent delete actions inside configuration.
- **Pros**: Provides a more complete project/session lifecycle.
- **Cons**: Larger UX and product scope; destructive actions require more careful confirmation and recovery requirements.
- **Why rejected**: Too broad for the selected MVP.

## Consequences

### Positive

- Keeps the PRD focused on the highest-frequency user expectation: returning to the same projects and sessions after restart.
- Reduces product ambiguity around archive and delete semantics.
- Allows stale project handling to ship without requiring a full lifecycle management model.

### Negative

- Users cannot archive active projects or sessions in V1.
- Users cannot delete valid project/session records through configuration in V1.
- Additional project/session cleanup workflows will need a later PRD or PRD update.

### Risks

- **Scope dissatisfaction**: Users who expect complete lifecycle management may find V1 incomplete. Mitigation: list archive/delete workflows as explicit Phase 2 candidates.
- **Stale list clutter**: Keeping stale projects visible may create noise. Mitigation: include remove action for stale projects in V1.

## Implementation Notes

This ADR describes product scope only. Technical implementation details belong in the follow-on TechSpec.
