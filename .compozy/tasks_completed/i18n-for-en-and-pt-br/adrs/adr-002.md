# ADR-002: Use Complete UI Parity With In-App Language Selection

## Status

Accepted

## Date

2026-05-08

## Context

The PRD clarification phase confirmed that V1 should promise complete app-owned UI parity across English and Brazilian Portuguese. The user also selected a visible in-app language selector rather than relying only on the operating system's app language preference. The main success signal for V1 is quality: zero launch-blocking localization defects across the V1 surfaces, validated first by an internal bilingual reviewer.

The product must preserve the boundary from ADR-001: app-authored UI is localizable, while user prompts, assistant output, paths, model IDs, provider names, raw terminal output, RPC payloads, and tool content remain verbatim.

## Decision

The PRD will use the Complete Parity + In-App Selector approach. V1 will provide complete app-owned UI localization in English and Brazilian Portuguese, expose a visible app language selector for users, preserve technical/user/generated content verbatim, and use internal bilingual review as the launch quality gate.

## Alternatives Considered

### Alternative 1: Complete Parity + OS Language Only

- **Description**: Localize the full app-owned UI but rely on macOS language preferences for language selection.
- **Pros**: Lower UI scope and closer to platform convention.
- **Cons**: Less discoverable for bilingual users and harder to validate language switching during app use.
- **Why rejected**: The user explicitly preferred a visible in-app language selector.

### Alternative 2: Critical Flow Pilot + Selector

- **Description**: Localize only critical flows first, paired with an in-app language selector.
- **Pros**: Faster release and lower review burden.
- **Cons**: Conflicts with the complete parity promise and risks mixed-language trust problems.
- **Why rejected**: The user selected complete app-owned UI parity as the V1 promise.

## Consequences

### Positive

- Gives users explicit control over app language.
- Supports a clear product claim that English and pt-BR are both first-class app UI languages.
- Makes bilingual review and quality validation easier because reviewers can switch language intentionally.

### Negative

- Adds language preference UX to the V1 scope.
- Increases QA coverage because both language selection and localized surfaces must be checked.
- Requires clear product copy so users understand that app language does not translate prompts, assistant output, logs, or other verbatim technical artifacts.

### Risks

- **User confusion about what changes language**: Mitigate with clear language-setting copy and release notes explaining the verbatim-content boundary.
- **Incomplete parity despite selector visibility**: Mitigate with an inventory-based quality gate and internal bilingual review.
- **Preference mismatch with OS language**: Mitigate by making the selected app language visibly understandable and reversible.

## Implementation Notes

- The PRD must describe the language selector as a user capability and avoid prescribing implementation details.
- The TechSpec should decide the exact preference model and platform integration details.
- V1 acceptance should treat missing or incorrect localization in included app-owned surfaces as launch-blocking.
