# ADR-001: Use SQLite for Project and Session Continuity

## Status

Accepted

## Date

2026-05-08

## Context

Pi Agent Native currently persists native project and session sidebar state as a JSON file at `~/Library/Application Support/PiAgentNative/sessions.json`. That state supports reopened projects and known Pi sessions between launches, but the V1 idea requires clearer local ownership: projects need stable local identity, path, name, and status; sessions need a local database session id, a separate Pi RPC session id generated by Pi RPC, title, status, project relation, and session file metadata.

Research shows recent projects and session restore are table-stakes behavior in IDEs and agent tools. SQLite is a credible local application-state substrate for desktop apps, but it introduces migration, schema, corruption, and stale-path responsibilities. The council agreed that the feature should be framed as a reliability move for end users, not as a generalized persistence platform.

## Decision

Use a local SQLite database for V1 project/session continuity. The SQLite store should remain narrowly scoped to recent/opened projects and per-project sessions, preserving the current user-facing restore behavior while clarifying durable identity and project-session ownership.

The local database id is the source of truth for native session identity. The Pi RPC session id is stored as runtime-linked metadata and must not be used as the primary local identity. Project paths must be canonicalized and validated before restore or association. The V1 durability contract must define project identity, session-to-project mapping, stale path handling, migration from JSON, duplicate prevention, and recoverable failure behavior.

## Alternatives Considered

### Alternative 1: Keep JSON Persistence

- **Description**: Continue storing all project/session state in `sessions.json`.
- **Pros**: Fastest delivery, no new dependency or schema migration.
- **Cons**: Keeps identity and lifecycle rules coupled to a whole-file Codable shape; weaker uniqueness, partial update, migration, and corruption handling.
- **Why rejected**: V1 specifically depends on reliable project/session continuity and separate local vs Pi RPC session identity.

### Alternative 2: General Local Persistence Platform

- **Description**: Use the SQLite migration as the start of a broad app persistence platform for preferences, search, cleanup, analytics, and future entities.
- **Pros**: Creates a foundation for future local-first capabilities.
- **Cons**: Expands scope beyond the user-facing MVP and delays validation of the restore workflow.
- **Why rejected**: The highest-value V1 is continuity reliability, not a platform rewrite.

### Alternative 3: Wrap JSON Behind a Repository Interface

- **Description**: Clarify the store contract but keep JSON as the underlying format.
- **Pros**: Improves boundaries with lower immediate implementation cost.
- **Cons**: Does not solve atomicity, queryability, uniqueness, or schema evolution pressure.
- **Why rejected**: It is a useful transitional shape, but insufficient as the durable V1 persistence substrate.

## Consequences

### Positive

- Stable local identity for projects and sessions.
- Clear separation between native session identity and Pi RPC session identity.
- Better support for atomic writes, uniqueness constraints, migrations, and future bounded queries.
- More reliable app restart behavior for recent projects and per-project sessions.

### Negative

- Requires schema design, migration tests, and backup or recovery behavior.
- Adds a new local artifact and possible SQLite sidecar files depending on journal mode.
- Requires explicit handling for stale paths, missing session files, and corrupt database state.

### Risks

- **Over-scoping**: Search, cleanup automation, sync, preferences, or analytics could turn V1 into a platform project. Mitigation: keep V1 limited to projects, sessions, selected project, and selected session restore.
- **Wrong restore target**: Incorrect identity mapping could associate a Pi session with the wrong local project. Mitigation: separate local ids from Pi RPC ids and validate project/session relationships before restore.
- **Data loss during migration**: JSON import could corrupt or drop existing sidebar state. Mitigation: keep a backup of the JSON source and provide recoverable failure behavior.
- **Sensitive metadata exposure**: Project paths, names, titles, and session files can reveal private work context. Mitigation: store locally in the app support directory with restrictive file handling and avoid expanding V1 into broad search/export.

## Implementation Notes

- Preserve the current `SessionStore` entry points initially so `AppModel` and UI-facing stores can migrate incrementally.
- Use separate `projects` and `sessions` records with a project foreign key.
- Include `project.status` and `session.status` in V1.
- Store both `session.id` as local database identity and `session.piSessionID` as Pi RPC metadata.
- Migrate existing `sessions.json` once when no SQLite database exists, and leave the source JSON recoverable.
- Treat stale project paths and missing session files as explicit states rather than silently deleting or relinking records.
