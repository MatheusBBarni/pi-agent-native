# ADR-001: Keep Subscription Login Provider-CLI Mediated

## Status

Accepted

## Date

2026-05-08

## Context

Pi Agent Native already exposes subscription login through the Login modal by running the provider login process, detecting provider login URLs from process output, opening those URLs in the browser, and refreshing access after the process completes. The desired user experience is that the modal starts login, opens the provider URL, lets the user authenticate with an existing Codex, Claude Code, or Copilot subscription/account, and confirms usable access in Pi Agent Native afterward.

The app does not sell, manage, or validate provider subscriptions directly. Provider CLIs and Pi credentials remain the authority for account state and subscription-backed capability. The current app also has no registered custom URL scheme or app-level redirect handler, so a native browser redirect back into Pi Agent Native is not an owned product contract today.

## Decision

V1 will keep subscription login provider-CLI mediated. Pi Agent Native will improve the Login modal handoff by reliably detecting provider login URLs, automatically opening only expected provider login URLs, keeping a manual Open Link fallback visible, and confirming login only after provider process completion plus Access Refresh.

V1 will not add native custom URL scheme redirect handling, billing-provider subscription management, or direct OAuth/session ownership. If the browser or provider returns control to the provider CLI, Pi Agent Native will reflect that through process output, exit status, RPC restart, and refreshed access state.

## Alternatives Considered

### Alternative 1: Add Native Redirect Handling in V1

- **Description**: Register a Pi Agent Native URL scheme or app-claimed URL, handle provider redirects through `onOpenURL`, and use browser redirect events as a first-class login completion path.
- **Pros**: Could make login feel more native and give the app a direct browser-return event.
- **Cons**: Creates a new app-owned authentication boundary, requires provider-specific redirect contracts, state/nonce validation, replay protection, callback parsing, and long-term maintenance.
- **Why rejected**: The current provider login commands already own authentication. Adding redirect handling before Pi Agent Native intentionally owns OAuth/session orchestration would increase security and maintenance scope without proving it is necessary for the reported V1 problem.

### Alternative 2: Open Any Detected Web URL Automatically

- **Description**: Continue opening every detected `http` or `https` URL from provider process output.
- **Pros**: Lowest friction and minimal implementation complexity.
- **Cons**: A malicious or compromised process output could cause Pi Agent Native to launch phishing or unsafe URLs from a trusted login surface.
- **Why rejected**: The Login modal should reduce friction only for expected provider login destinations. Unknown URLs should require explicit user intent through a visible fallback.

## Consequences

### Positive

- Preserves the existing authority boundary: provider CLIs and Pi credentials own authentication.
- Focuses V1 on the reported failure: browser handoff and post-login confirmation.
- Keeps the implementation reversible and testable through process output, URL detection, duplicate suppression, and Access Refresh.
- Reduces credential-phishing risk by avoiding arbitrary URL auto-open behavior.

### Negative

- Pi Agent Native will not have a native browser redirect callback in V1.
- Login success remains indirect and depends on provider process completion plus Access Refresh.
- Some provider-specific edge cases may still require terminal output or manual fallback handling.

### Risks

- **Provider output changes**: URL detection may miss new CLI output formats. Mitigate with accumulated-output parsing tests and provider-specific fixtures.
- **False success**: Process exit alone may not prove usable access. Mitigate by requiring Access Refresh before showing confirmed access.
- **Blocked browser open**: `NSWorkspace` or the provider URL may fail to open. Mitigate with a persistent Open Link fallback and clear modal state.
- **Unsafe URL launch**: Process output may contain unexpected URLs. Mitigate with provider-domain allowlisting for automatic opening.

## Implementation Notes

- Keep Provider Login URL as the domain term for URLs emitted by provider login processes.
- Model the user-visible states as provider flow started, browser link available/opened, waiting for provider completion, refreshing access, access confirmed, and failed.
- Browser-open success must not be treated as authentication success.
- Access Refresh remains the only V1 confirmation that subscription-backed model access is usable.
