# ADR-002: Prioritize Provider Login State Clarity

## Status

Accepted

## Date

2026-05-08

## Context

The PRD clarification identified power users who switch between subscription providers as the primary MVP audience. The MVP should cover all current subscription providers shown in the Login modal and should optimize for clear login progress and confirmation states rather than expanding subscription management or building a full account hub.

Provider switching creates a trust problem when users cannot tell whether the selected provider is starting login, waiting for browser/provider completion, refreshing access, confirmed, failed, or stopped. The app must avoid making stale access from one provider look like confirmed access for another provider.

## Decision

The PRD will use the Provider State Clarity approach for MVP. Pi Agent Native will make the Login modal expose a clear state sequence for each subscription login attempt: starting, waiting for provider/browser completion, refreshing access, confirmed, failed, and stopped. The modal will keep all existing subscription providers in scope.

This approach will focus on what users need to understand and trust during provider switching. It will not turn the Login modal into a full account management center.

## Alternatives Considered

### Alternative 1: Provider Switching Guardrails

- **Description**: Focus mainly on preventing stale access from appearing active after provider changes.
- **Pros**: Strong safety value and smaller MVP surface.
- **Cons**: Does not fully solve the user's need to see current login progress and confirmation.
- **Why rejected**: It is useful but too narrow for the selected MVP outcome.

### Alternative 2: Full Login Control Center

- **Description**: Add richer provider status, troubleshooting, provider history, and account-switch guidance.
- **Pros**: More comprehensive provider management experience.
- **Cons**: Larger scope, more copy and state design, and higher risk of implying Pi Agent Native manages provider accounts or subscriptions.
- **Why rejected**: Too broad for MVP and misaligned with the provider-CLI ownership boundary.

## Consequences

### Positive

- Gives provider-switching users clear progress and confirmation states.
- Reduces confusion caused by stale or ambiguous access state.
- Keeps all current subscription providers in scope without adding billing or account management.
- Aligns with ADR-001 by treating provider login as provider-CLI mediated.

### Negative

- Does not provide a full provider account dashboard.
- Does not add provider-specific subscription details beyond confirmed usable access.
- Requires careful copy so users understand Pi Agent Native verifies access but does not manage subscription plans.

### Risks

- Users may still expect direct subscription plan details. Mitigate with clear wording that providers own account and subscription status.
- Some provider flows may not expose enough signal for fine-grained state. Mitigate by showing honest fallback states and preserving process output for troubleshooting.
- Confirmation may be delayed by Access Refresh. Mitigate by making refreshing state explicit.

## Implementation Notes

- Product states should be described in user-facing language, not implementation terminology.
- Browser-open success is not confirmation.
- Confirmed state means the latest access check indicates usable provider-backed model access.
