# ADR-003: Own Subscription Login Attempt State in OAuthLoginRunner

## Status

Accepted

## Date

2026-05-08

## Context

The PRD requires the Login modal to show a clear provider login state sequence for all current subscription providers. The existing implementation already has `OAuthLoginRunner` as the process boundary for provider login commands, terminal output, provider URL detection, current provider, attempt ID, and exit status. `AppModel` coordinates PiRPC restart and Access Refresh after login completion.

The user clarified that the login flow should behave like `pi` in a terminal: a successful provider login or provider switch must preserve the current session, and model/reasoning selection should continue when the refreshed provider still supports the selected model. Confirmation should come from the existing PiRPC `get_state` and `get_available_models` refresh path for MVP.

## Decision

`OAuthLoginRunner` will own the structured, user-facing subscription login attempt state. It will expose the active provider, attempt ID, latest Provider Login URL, process output, and a login phase such as not started, starting, waiting for provider, refreshing access, confirmed, failed, and stopped.

`AppModel` will remain responsible for PiRPC coordination. It will update the runner-owned state when provider login starts, stops, exits, begins Access Refresh, and receives Access Refresh completion or failure. Confirmation will be derived from existing PiRPC refresh responses, not from browser-open success.

The current session will be preserved after successful provider login or provider switch. The selected model and reasoning setting will be preserved when refreshed model availability still supports them; otherwise the app will require the user to pick a valid model.

## Alternatives Considered

### Alternative 1: AppModel Owns a Separate Subscription Login State

- **Description**: Add a separate `@Published subscriptionLoginAttempt` value to `AppModel`, while keeping `OAuthLoginRunner` process-only.
- **Pros**: Keeps process execution and UI state separate, and centralizes application state in `AppModel`.
- **Cons**: Splits one login attempt across two state owners and requires duplicated provider, attempt ID, URL, and exit-status coordination.
- **Why rejected**: The user selected runner-owned state because the runner already represents the terminal-like provider login process.

### Alternative 2: Derive State Only in LoginSheetView

- **Description**: Keep existing models unchanged and compute modal copy directly from `oauthRunner`, `authAccess`, and selected provider.
- **Pros**: Smallest code change.
- **Cons**: Pushes state interpretation into the view, makes lifecycle testing weaker, and risks stale provider-specific copy.
- **Why rejected**: The modal needs a testable state sequence that is not encoded as ad hoc view logic.

### Alternative 3: Add a New PiRPC Access-Status Command in MVP

- **Description**: Extend PiRPC with a new command that directly returns authenticated provider and subscription-backed access.
- **Pros**: Would provide a cleaner confirmation contract than inferring from current refresh responses.
- **Cons**: Expands protocol scope and requires upstream PiRPC changes before the Login modal improvement can ship.
- **Why rejected**: Existing `get_state` and `get_available_models` refresh responses are sufficient for MVP confirmation.

## Consequences

### Positive

- Keeps terminal-like provider login state close to the process that produces it.
- Gives the Login modal one structured state source instead of interpreting raw output.
- Preserves the existing provider-CLI and PiRPC authority boundaries.
- Allows unit tests to cover the runner-owned phase transitions without requiring SwiftUI view tests.

### Negative

- `OAuthLoginRunner` will contain more than process execution state.
- `AppModel` must explicitly feed PiRPC refresh outcomes back into the runner-owned state.
- Future PiRPC access-status work may require adapting the confirmation bridge.

### Risks

- **State drift**: Runner state and `authAccess` could disagree. Mitigate by letting AppModel be the only bridge from Access Refresh to runner confirmation.
- **False confirmation**: Login process exit may be mistaken for confirmed access. Mitigate by reserving confirmed phase for successful PiRPC refresh with usable provider-backed model access.
- **Model mismatch after provider switch**: Current model may not exist after refresh. Mitigate by preserving it only when refreshed availability still supports it and otherwise requiring explicit user selection.

## Implementation Notes

- Browser-open success must transition at most to a waiting phase, never to confirmed.
- Stopped and failed phases must retain enough provider context for the modal to explain what happened.
- Unit tests should cover runner phase transitions and AppModel refresh-to-runner confirmation.
- Manual verification should cover the rendered Login modal states.
