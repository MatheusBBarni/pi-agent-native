# ADR-003: Use Assignment-Time Localization With Native Strings Resources

## Status

Accepted

## Date

2026-05-08

## Context

Pi Agent Native currently has no localization resources or lookup layer. App-owned strings are spread across SwiftUI views and model-owned surfaces such as `AppModel`, `DefaultKeymap`, `AuthAccessState`, command palette items, process logs, and status text. The PRD requires complete app-owned English and pt-BR UI parity plus a visible language selector in the configuration modal.

Runtime language switching creates a choice: store localized display strings when app-owned status/log/message state is assigned, or store localization keys/value objects so existing app-owned state can re-render immediately when language changes.

## Decision

Use native `.strings` and `.stringsdict` resources for English and Brazilian Portuguese, backed by a small localization facade. Persist the selected app language through the existing `UserDefaults`/`@Published` settings pattern.

For V1, localize model-owned display strings at assignment time. Already-shown app-owned statuses, logs, or messages may remain in the previous language until they update naturally. Technical and generated content remains verbatim and is never passed through localization.

## Alternatives Considered

### Alternative 1: Localization Keys and Value Objects

- **Description**: Store app-owned status/help/log state as localization keys with interpolation values, then resolve them at render time.
- **Pros**: Existing visible app-owned state re-renders immediately after language changes.
- **Cons**: More invasive across current string-heavy model state and tests.
- **Why rejected**: V1 should keep implementation scope proportional. Immediate re-rendering of older app-owned statuses is less important than complete parity for current UI surfaces.

### Alternative 2: Xcode String Catalogs

- **Description**: Use `.xcstrings` resources instead of `.strings` and `.stringsdict`.
- **Pros**: Better modern Xcode localization workflow.
- **Cons**: Adds tool and package workflow uncertainty for this SwiftPM-first repository.
- **Why rejected**: The selected target format is native `.strings`/`.stringsdict`, which is straightforward for SwiftPM resources and manual app bundling.

## Consequences

### Positive

- Fits the current SwiftPM project and avoids a broad model-state refactor.
- Keeps localization explicit for app-owned UI while preserving verbatim technical artifacts.
- Supports pluralization and formatted strings with `.stringsdict` where needed.

### Negative

- Some existing app-owned status/log strings can remain in the previous language after switching until they update.
- Tests that assert exact English strings must be updated carefully.
- Future work may need a key/value state model if runtime switching quality expectations increase.

### Risks

- **Mixed-language app-owned stale state**: Mitigate by documenting the V1 behavior and refreshing high-visibility settings-controlled UI immediately.
- **Accidental localization of technical content**: Mitigate by keeping technical values outside localization helpers unless they are interpolation values.
- **Resource lookup regressions**: Mitigate with unit tests for both supported languages.

## Implementation Notes

- Add `AppLanguage` and a small localization facade under the existing core target.
- Add English and pt-BR `.lproj` resources under the core target's processed resources.
- Persist the language selection with a dedicated `UserDefaults` key.
- Apply the selected locale to SwiftUI rendering and use the facade for model-owned strings.
