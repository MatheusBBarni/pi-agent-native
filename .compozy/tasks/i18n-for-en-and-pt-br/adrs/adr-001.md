# ADR-001: Scope English and Brazilian Portuguese UI Localization

## Status

Accepted

## Date

2026-05-07

## Context

Pi Agent Native is a SwiftPM macOS SwiftUI shell for the `pi` coding agent. The app currently has no localization resources, no localization API usage, and many hardcoded English user-facing strings across menus, sidebar, chat, login, settings, inspector, process log, extension dialogs, keybinding help, status and error text, accessibility labels, and tests.

The feature idea is to support English and Brazilian Portuguese equally for the app UI. Research shows Brazil is a large digital market, and market data supports user preference for native-language product experiences. The council agreed that localization is valuable, but only if the V1 boundary prevents ambiguity between localized app-authored UI and verbatim content produced by users, models, tools, files, paths, providers, or RPC/terminal output.

## Decision

V1 will localize complete app-authored, end-user UI for English and Brazilian Portuguese. This includes app chrome, menus, controls, settings, login/auth states, help text, empty states, accessibility strings, app-owned errors, and app-owned status messages.

V1 will not localize user prompts, assistant output, project names, file paths, model IDs, provider names, external app names, raw RPC payloads, raw terminal output, or generated/tool content. These artifacts remain verbatim for trust, debugging, and supportability.

The V1 definition of "complete" must be inventory-based: a known set of included surfaces and core journeys must be verified in both locales. Translation tables alone are not sufficient.

## Alternatives Considered

### Alternative 1: Minimal Partial Translation

- **Description**: Translate only the most visible controls, such as navigation, settings, and the main chat chrome.
- **Pros**: Lower initial effort, faster validation, less test churn.
- **Cons**: Creates mixed-language flows, weakens trust in auth/error states, and makes pt-BR support feel unfinished.
- **Why rejected**: The user selected complete V1, and the council found partial localization risky for a coding-agent shell where state, consent, and errors matter.

### Alternative 2: Full Localization Platform

- **Description**: Build a broader localization platform with translation management, future-language workflows, runtime switching, and generated typed key infrastructure.
- **Pros**: Scales better for many locales and larger teams.
- **Cons**: Adds infrastructure before there is evidence of multi-locale scale and may delay core agent usability work.
- **Why rejected**: V1 only needs English and pt-BR. Native Apple localization plus clear conventions is sufficient until scale pressure justifies custom infrastructure.

### Alternative 3: Content Localization

- **Description**: Translate assistant output, prompts, tool logs, file paths, provider names, and other generated or external content.
- **Pros**: Could make the entire session appear more native to pt-BR users.
- **Cons**: Blurs trust boundaries, harms debugging, can alter technical meaning, and creates security/support ambiguity.
- **Why rejected**: The council agreed that generated/user/system artifacts must stay verbatim in V1.

## Consequences

### Positive

- Gives English and Brazilian Portuguese users equal access to app-owned UI.
- Establishes localization as a UI contract before more hardcoded strings accumulate.
- Preserves debugging and trust by keeping external/generated artifacts verbatim.
- Enables measurable validation of pt-BR market value without building a full localization platform.

### Negative

- Requires broad string inventory and migration across existing UI and tests.
- Increases QA scope because both locales must be checked for missing strings, clipping, truncation, and critical-flow clarity.
- Introduces ongoing translation ownership for all future app-owned UI strings.

### Risks

- **Mixed-language critical paths**: Mitigate with an inventory of included surfaces and launch-blocking checks for auth, consent, errors, and core flows.
- **Unclear localization boundary**: Mitigate with explicit "do not localize" rules for user/generated/system artifacts.
- **Translation quality issues**: Mitigate with reviewed terminology for security-sensitive and workflow-critical strings.
- **Bundle/resource regressions**: Mitigate by verifying SwiftPM and app bundle builds load localization resources.

## Implementation Notes

- Prefer Apple's native localization path for SwiftUI and macOS, including resource-backed strings and String Catalogs or equivalent package resources.
- Add lightweight coverage checks that detect missing English and pt-BR strings in V1 surfaces.
- Update tests that assert English UI copy so they verify stable behavior without blocking localization.
- Validate core flows in both locales: first-run/no-project state, project selection, chat composer, model/thinking controls, login/auth, settings, inspector, process log, keybinding help, extension dialogs, and error states.
