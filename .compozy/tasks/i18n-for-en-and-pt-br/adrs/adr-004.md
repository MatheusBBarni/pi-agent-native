# ADR-004: Use Warning-Based Coverage Checks and SwiftPM Resource Bundle Packaging

## Status

Accepted

## Date

2026-05-08

## Context

The PRD sets quality as the main success signal, but the user selected warning-based automated coverage rather than failing tests for every missing required localization key. The repository is a SwiftPM macOS app with a manual `.app` bundling script. Today, the script copies only the executable and icon, so processed localization resources must be included explicitly.

The test strategy must prioritize useful automated coverage without turning the first localization pass into full UI snapshot infrastructure.

## Decision

Automated localization coverage will emit warnings for missing required English or pt-BR keys. Maintainer review determines release readiness.

The manual app bundle script will copy the SwiftPM-generated resource bundle into `Contents/Resources` so packaged app builds can load localization resources consistently.

V1 tests will prioritize unit tests for localization lookup, language persistence, required-key warnings, and representative model-owned app strings. Snapshot/UI tests for every localized screen are out of scope for V1.

## Alternatives Considered

### Alternative 1: Fail Tests on Any Missing Key

- **Description**: Make missing English or pt-BR keys fail automated tests.
- **Pros**: Strongest automatic coverage enforcement.
- **Cons**: Too rigid for the initial migration and can block iterative review before translations are final.
- **Why rejected**: The user selected warnings with maintainer review deciding release readiness.

### Alternative 2: Snapshot or UI Tests for Every Localized Screen

- **Description**: Add visual or UI automation coverage for every localized surface.
- **Pros**: Better detection of clipping, truncation, and missing strings.
- **Cons**: High setup and maintenance cost for V1.
- **Why rejected**: V1 should prioritize unit-level correctness and maintainer review. Full screenshot/UI coverage can be added later if localization scope expands.

### Alternative 3: Copy Raw `.lproj` Directories Directly

- **Description**: Copy raw localization directories into the generated app bundle manually.
- **Pros**: Simple to reason about for plain resources.
- **Cons**: Can diverge from SwiftPM's processed resource layout and miss bundle expectations.
- **Why rejected**: Copying the SwiftPM resource bundle better matches how the executable will resolve `Bundle.module` resources.

## Consequences

### Positive

- Keeps V1 test scope focused and maintainable.
- Preserves maintainer judgment for translation completeness and terminology quality.
- Ensures packaged app builds can access localization resources.

### Negative

- Missing-key warnings require human discipline and can be ignored.
- Manual QA remains necessary for layout, truncation, and quality review.
- Resource bundle copying needs careful build-script verification across debug and release builds.

### Risks

- **Warnings missed before release**: Mitigate with a documented release checklist and maintainer sign-off.
- **Packaged app lacks resources**: Mitigate with a build-script verification test or manual check.
- **Layout defects escape unit tests**: Mitigate with manual bilingual QA for V1 surfaces.

## Implementation Notes

- Add unit tests that exercise lookup for `en` and `pt-BR`.
- Add a coverage reporter that returns warnings rather than failing by default.
- Update `Scripts/build-app.sh` to copy the generated SwiftPM resource bundle into `Contents/Resources`.
- Add generated `Info.plist` localization metadata if needed for macOS language discovery.
